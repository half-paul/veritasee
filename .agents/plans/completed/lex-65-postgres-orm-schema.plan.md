# Plan: Provision managed Postgres + ORM + initial schema

## Summary

Stand up the data layer for Veritasee Override: a managed **Neon** Postgres database, a shared **Drizzle ORM** package at `packages/db`, and a first migration that materialises the PRD §8 schema (`users`, `articles`, `snapshots`, `corrections`, `references`, `ai_runs`, `moderation_decisions`, `reputation_events`) plus the `pgvector` extension. The decision (Drizzle over Prisma, Neon over Supabase) is captured in ADR 0002 before any code lands. A small migration test asserts `pgvector` is loadable so later evidence-search work has a known foundation. Quotas remain in Redis (out of scope for this issue).

## Decision Rationale (for ADR 0002)

**ORM — Drizzle over Prisma**
- TS-first, SQL-shaped schema fits the strict `verbatimModuleSyntax` + `noUncheckedIndexedAccess` config in `tsconfig.base.json`; no codegen step before each typecheck.
- `pgvector` has first-class community support via `customType` in `drizzle-orm/pg-core`; Prisma still requires `Unsupported("vector")` workarounds.
- Migrations are plain SQL files generated by `drizzle-kit`, easy to read and review per Vercel deployment.
- Lighter cold-start and bundle size on Vercel/edge than Prisma's generated client.

**Provider — Neon over Supabase**
- Serverless Postgres with per-branch databases — preview deployments can spin up isolated DB branches via Vercel/Neon integration.
- Native `pgvector` (no extension juggling) and an HTTP driver (`@neondatabase/serverless`) suitable for Edge Runtime if we move that direction.
- Auth is already Clerk (ADR 0001); Supabase's bundled auth would be unused overlap.

**Tradeoffs accepted**
- Drizzle has fewer "batteries included" features than Prisma Studio/Accelerate; we accept this — query inspection happens via Neon console + ad-hoc psql.
- Neon's free tier has a connection limit; we mitigate via the pooled connection string for app traffic and the direct URL only for migrations.

## PRD §8 Schema (verbatim, source of truth)

> Source: `docs/PRD.md:209-219`
>
> - `users(id, role, trust_points, byok_provider, byok_key_encrypted, ...)`
> - `articles(id, source_url, source_domain, current_revision_hash, topic_tags[], last_fetched_at)`
> - `snapshots(id, article_id, revision_hash, content, fetched_at)`
> - `corrections(id, article_id, snapshot_id, anchor_text_fragment, anchor_prefix, anchor_suffix, body_md, verity_score, rationale, status, author_id, created_at)`
> - `references(id, correction_id, url_or_identifier, title, author, published_at, accessed_at)`
> - `ai_runs(id, correction_id, provider, model, scenario, tokens_in, tokens_out, cost_usd, evidence_json, created_at)`
> - `moderation_decisions(id, correction_id, moderator_id, decision, reason, created_at)`
> - `reputation_events(id, user_id, delta, reason, correction_id, created_at)`
> - `quotas` lives in Redis, not Postgres.

### Concrete column shapes (filled where PRD uses `...`)

These choices align with PRD §3 (RBAC), §5 (anchors, AI scenarios, moderation), and ADR 0001 (Clerk `external_id`).

- **`users`**: `id uuid pk default gen_random_uuid()`, `external_id text unique not null` (Clerk userId — see ADR 0001 §"Migration Path"), `email text unique not null`, `role text not null default 'contributor'` constrained to `('reader','contributor','moderator','admin')` to match `Role` in `apps/web/src/lib/auth/roles.ts`, `trust_points integer not null default 0`, `byok_provider text` constrained to `('anthropic','openai','gemini','openrouter')` nullable, `byok_key_encrypted text` nullable, `created_at timestamptz not null default now()`, `updated_at timestamptz not null default now()`.
- **`articles`**: `id uuid pk`, `source_url text not null`, `source_domain text not null`, `current_revision_hash text`, `topic_tags text[] not null default '{}'`, `last_fetched_at timestamptz`, unique `(source_url)`.
- **`snapshots`**: `id uuid pk`, `article_id uuid not null references articles(id) on delete cascade`, `revision_hash text not null`, `content text not null`, `fetched_at timestamptz not null default now()`, unique `(article_id, revision_hash)`.
- **`corrections`**: `id uuid pk`, `article_id uuid not null references articles(id)`, `snapshot_id uuid not null references snapshots(id)`, `anchor_text_fragment text not null`, `anchor_prefix text`, `anchor_suffix text`, `body_md text not null`, `verity_score integer` (0–100, check constraint), `rationale text`, `status text not null default 'pending'` constrained to `('pending','approved','rejected','withdrawn')`, `author_id uuid not null references users(id)`, `created_at timestamptz not null default now()`.
- **`references`** (table name `correction_references` — `references` is a reserved word in SQL/PG; we keep the PRD field labels but rename the relation): `id uuid pk`, `correction_id uuid not null references corrections(id) on delete cascade`, `url_or_identifier text not null`, `title text`, `author text`, `published_at timestamptz`, `accessed_at timestamptz not null default now()`.
- **`ai_runs`**: `id uuid pk`, `correction_id uuid references corrections(id) on delete set null`, `provider text not null`, `model text not null`, `scenario text not null` constrained to `('quick','academic','adversarial')` per PRD §5, `tokens_in integer not null default 0`, `tokens_out integer not null default 0`, `cost_usd numeric(10,4) not null default 0`, `evidence_json jsonb not null default '{}'::jsonb`, `created_at timestamptz not null default now()`.
- **`moderation_decisions`**: `id uuid pk`, `correction_id uuid not null references corrections(id) on delete cascade`, `moderator_id uuid not null references users(id)`, `decision text not null` constrained to `('approve','reject','revise')`, `reason text`, `created_at timestamptz not null default now()`.
- **`reputation_events`**: `id uuid pk`, `user_id uuid not null references users(id) on delete cascade`, `delta integer not null`, `reason text not null`, `correction_id uuid references corrections(id) on delete set null`, `created_at timestamptz not null default now()`.

Indexes (initial, conservative): `articles(source_domain)`, `corrections(article_id, status)`, `corrections(author_id)`, `ai_runs(correction_id)`, `reputation_events(user_id, created_at)`.

`pgvector`: created via `CREATE EXTENSION IF NOT EXISTS vector;` in the first migration. No vector column ships in this issue — it's enabled and asserted loadable so later evidence-search work doesn't have to migrate again.

## Patterns to Follow

The repo has no prior DB code. Patterns below come from the project's existing conventions and Drizzle's official Next.js/Neon recipe (current as of 2026-04).

### Workspace package shape (mirror `packages/` convention)

```jsonc
// SOURCE: target — packages/db/package.json (no prior package; mirror pnpm workspace style from /package.json:1-22)
{
  "name": "@veritasee/db",
  "version": "0.0.0",
  "private": true,
  "type": "module",
  "main": "./src/index.ts",
  "types": "./src/index.ts",
  "scripts": {
    "lint": "eslint .",
    "typecheck": "tsc --noEmit",
    "db:generate": "drizzle-kit generate",
    "db:migrate": "tsx scripts/migrate.ts",
    "db:studio": "drizzle-kit studio"
  }
}
```

### Strict TS in shared packages (mirror `tsconfig.base.json`)

```jsonc
// SOURCE: target — packages/db/tsconfig.json; extend repo base
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "outDir": "dist",
    "rootDir": "src",
    "moduleResolution": "bundler"
  },
  "include": ["src/**/*", "drizzle.config.ts", "scripts/**/*"]
}
```

### Role union mirrors auth helper (single source of truth across packages)

```ts
// SOURCE: apps/web/src/lib/auth/roles.ts:1
export type Role = 'reader' | 'contributor' | 'moderator' | 'admin';
```

The DB `users.role` check constraint must use exactly these four values. The Drizzle column type uses `text` + check, not `pgEnum`, so we can change the union without an ALTER TYPE migration.

### Env wiring (mirror `.env.example` style)

```bash
# SOURCE: apps/web/.env.example:1-19 — extend in same file
# Postgres (Neon) — pooled URL for app traffic, direct URL for migrations
DATABASE_URL=postgres://...neon.tech/veritasee?sslmode=require&pgbouncer=true
DATABASE_URL_UNPOOLED=postgres://...neon.tech/veritasee?sslmode=require
```

### Validation pattern for env at boundary (mirror `resolveRole` defensiveness)

```ts
// SOURCE: apps/web/src/lib/auth/roles.ts:9-15 (defensive narrowing pattern)
function requireEnv(name: string): string {
  const v = process.env[name];
  if (!v) throw new Error(`Missing required env: ${name}`);
  return v;
}
```

## Risks

| Risk | Mitigation |
|------|------------|
| Reserved-word collision: PRD's `references` table conflicts with SQL keyword | Rename relation to `correction_references` while keeping PRD field labels unchanged. Document in ADR. |
| Neon free-tier connection limits exhaust under preview branches | Use pooled URL for app traffic; unpooled only for migrations. Connection count stays bounded. |
| `pgvector` not enabled on Neon free tier or region | Verified available on all Neon tiers and regions (2026-04). If a future region drops support, migration test fails loudly — that's the point of Task 8. |
| Drizzle schema drift vs hand-edited migrations | Treat `migrations/` as append-only after merge; never edit prior migrations. New schema changes always go through `drizzle-kit generate`. |
| `users.role` constraint diverges from `Role` union in `apps/web/src/lib/auth/roles.ts` | Both are `('reader','contributor','moderator','admin')`. Add a comment in the schema file pointing back to `roles.ts:1` so future edits propagate. |
| Migration test runs against shared dev branch and pollutes data | Test is read-only (`select` from `pg_extension`). No writes. |
| `DATABASE_URL` accidentally committed via `.env.local` | `.env.local` is already in standard Next.js gitignore. Verify before first commit. |
