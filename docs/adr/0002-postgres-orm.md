# ADR 0002 — Postgres + ORM: Neon + Drizzle

- **Status:** Accepted
- **Date:** 2026-04-29
- **Linear:** LEX-65

## Context

Veritasee Override needs a relational data store for the PRD §8 schema (`users`, `articles`, `snapshots`, `corrections`, `references`, `ai_runs`, `moderation_decisions`, `reputation_events`) and headroom for vector-backed evidence search later in v1. The v1 architecture in `docs/general/SYSTEM-OVERVIEW.md` mandates managed services; we will not run our own Postgres.

We need a Postgres provider and an ORM that:

1. Run cleanly under Vercel + Next.js 15 App Router (server actions and route handlers).
2. Support `pgvector` natively so evidence-search in a future issue does not require migrating providers.
3. Fit the repo's strict TS settings (`verbatimModuleSyntax`, `noUncheckedIndexedAccess`) without a heavyweight codegen step.
4. Allow per-branch databases for preview deployments so PRs can exercise schema changes in isolation.

## Decision

We adopt **Neon** as the managed Postgres provider and **Drizzle ORM** (`drizzle-orm` + `drizzle-kit`) as the schema/migration toolchain for v1.

- The shared data layer lives in a workspace package at `packages/db` exposing a `db` client and a `schema` namespace.
- The Neon HTTP driver (`@neondatabase/serverless`) is used at runtime; the same driver is used to apply migrations from a tiny `tsx` runner.
- Migrations are plain SQL files in `packages/db/migrations`, generated by `drizzle-kit generate`. They are append-only after merge.
- The first migration enables `pgvector` (`CREATE EXTENSION IF NOT EXISTS vector;`) so a later evidence-search column can be added without provisioning work.
- A pooled URL (`DATABASE_URL`, `pgbouncer=true`) is used for app traffic; a direct URL (`DATABASE_URL_UNPOOLED`) is used only for migrations and admin commands, which do not tolerate transaction-mode pooling.
- The `users.role` column is constrained to `('reader','contributor','moderator','admin')` to mirror the `Role` union in `apps/web/src/lib/auth/roles.ts`. The `users.external_id` column mirrors the Clerk userId per ADR 0001 §"Migration Path".

## Consequences

**Easier**

- TS-first ORM with SQL-shaped schema; no codegen step blocking each typecheck.
- `pgvector` works without extension juggling on Neon, on every region and tier (verified 2026-04).
- Per-branch databases make preview-deploy schema testing cheap.
- App and migrations share one driver, so connection-string handling is uniform.

**Harder / Constrained**

- Drizzle is younger than Prisma; query inspection relies on the Neon console and ad-hoc `psql` rather than a polished Studio.
- Neon's free tier caps connections; we mitigate via the pooled URL for app traffic.
- Switching providers later means re-pointing connection strings and revalidating that the destination supports `pgvector`. Schema files and migrations are vendor-neutral SQL, so the surface area is bounded.

## Cost Model

Snapshot dated 2026-04:

- **Neon free tier:** sufficient for closed-beta usage (compute autosuspends; storage well under cap).
- **Neon Launch tier:** ~$19/month; lifts compute hours and adds longer history.
- Migration overhead is zero after the first SQL is committed.

Closed-beta scope is well inside the free tier. We will revisit at v1.2 once contributor authoring opens to the public.

## Migration Path

If we move off Neon:

1. `pg_dump` the prod branch (managed export from the Neon console).
2. Stand up the replacement Postgres (RDS, Supabase, Render, etc.) with `pgvector` available.
3. Restore the dump, rewrite `DATABASE_URL` / `DATABASE_URL_UNPOOLED` in env, redeploy.
4. The Drizzle schema files and `migrations/*.sql` files do not change.

If we move off Drizzle:

1. Schema files become reference; migrations stay (they are plain SQL).
2. Replace `packages/db/src/client.ts` with the new ORM's client.
3. Domain code that imports inferred row types from `@veritasee/db` updates to the new ORM's equivalent.

## Alternatives Considered

### Prisma + Neon

Mature, batteries-included, generated client. Rejected for v1 because:

- `pgvector` requires `Unsupported("vector")` workarounds; we want first-class support so evidence search is straightforward.
- Codegen step adds a hop before each typecheck.
- Generated client is heavier on Vercel cold starts than Drizzle's tree-shaken output.

### Drizzle + Supabase

Viable. Rejected because:

- Supabase bundles auth, which we already get from Clerk (ADR 0001). Paying for two auth systems is wasteful even on free tiers.
- Neon's per-branch DB for preview deployments is more directly useful than Supabase's project model for our flow.

### Drizzle + Self-hosted RDS / Cloud SQL

Rejected: explicitly violates the "use managed services" principle in `docs/general/SYSTEM-OVERVIEW.md` and adds connection-pool, backup, and patching ops we do not want at v1.

## Schema Notes

- The PRD §8 `references` relation is renamed to `correction_references` in the database to avoid the SQL reserved word `REFERENCES`. PRD field labels are preserved.
- `quotas` are NOT in Postgres; they remain in Redis per PRD §8.
- `users.role` uses a `text` column with a `CHECK` constraint, not a `pgEnum`, so the union can be evolved without an `ALTER TYPE` migration.
- `verity_score` is `0..100` enforced via a `CHECK` constraint.
- `ai_runs.scenario` is constrained to `('quick','academic','adversarial')` per PRD §5.
- `moderation_decisions.decision` is constrained to `('approve','reject','revise')`.
- `corrections.status` is constrained to `('pending','approved','rejected','withdrawn')`.

## Security Posture

- No secrets land in source. `DATABASE_URL` and `DATABASE_URL_UNPOOLED` are sourced from env at boundaries via a `requireEnv()` helper that throws on missing values rather than silently using a default.
- BYOK material (`users.byok_key_encrypted`) is stored encrypted at rest at the application layer; the column never holds plaintext. The encryption key management plan is out of scope for this ADR and tracked separately.
- Migrations apply via the unpooled URL only, kept out of the pooled production traffic path.
- Per-branch DBs in preview deployments isolate experimental schema changes from production data.
